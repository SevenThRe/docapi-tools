## 1. 動作記述の原則

### 1.1 因果関係の明示

すべての動作は「トリガー → 結果」の形で書く。

**テンプレート:**
```
{ユーザー操作/システムイベント}の{時/場合}、{システムの応答}
```

**良い例:**
- 「上アイコンを押下する時、表示している社員の直前社員へ切替して表示する」
- 「全選択チェックオンの場合、全ての行選択もチェックオンとする」
- 「ホバーの場合、顔写真のファイル名、操作アイコンが表示される」

**悪い例:**
- 「社員を切り替える」 → いつ？どう？
- 「データを表示する」 → 何のデータ？どこに？
- 「処理を行う」 → 何の処理？

### 1.2 境界条件の明記

ループ・切替操作では、端の振る舞いを必ず書く。

```
上アイコンを押下する時、表示している社員の直前社員へ切替して表示する
先頭の社員の場合、末尾の社員へ切り替える  ← これが境界条件
下アイコンを押下する時、表示している社員の直後社員へ切替して表示する
末尾の社員の場合、先頭の社員へ切り替える  ← これも境界条件
```

### 1.3 条件分岐の列挙

複数条件がある場合、各条件を別行で対比的に書く:
```
ブロックモードの場合、サイドバーの全て社員
一覧モードの場合、サイドバーで選択された社員
```

条件と結果を同一行内に書き、条件を揃えることで対比を明確にする。

### 1.4 記述深度の判断基準

| UI要素の複雑度 | 記述行数目安 | 含めるべき定義 |
|--------------|------------|--------------|
| 単純表示（ラベル） | 1〜2行 | 種類 + 動作 |
| 単純操作（単体ボタン） | 3〜5行 | 種類 + 属性 + 動作 + 表示制御 |
| 標準操作（API連携あり） | 5〜8行 | 上記 + API + エラー |
| 複合操作（複数サブ要素） | 10行以上 | ネスト構造 + 各サブ要素の全定義 |

## 2. 表示制御の網羅パターン

### 2.1 チェックすべき4つの軸

すべてのインタラクティブ要素に対して、以下の4軸で表示制御を検討する:

**軸1: モード条件**
- 「ブロックモードの場合のみ表示する」
- 「一覧モードで複数人の場合は非活性とする」

**軸2: データ条件**
- 「顔写真がある場合のみ表示する」
- 「レスポンスの機能利用フラグがnoneの場合、非表示」

**軸3: 権限条件**
- 「起動権限ある機能のみ表示する」
- 「参照権限がない項目が非表示とする」

**軸4: 状態条件**
- 「社員1名選択している場合のみ活性」
- 「自分の顔写真がある場合のみ表示」

### 2.2 活性/非活性 vs 表示/非表示

使い分けのルール:
- **非表示**: その要素がそもそも不要な場合（権限なし、モードが異なる）
- **非活性**: 要素は見えるが操作できない場合（条件が揃っていない）
- **アイコン切替 + 非活性**: 要素のアイコンを変えた上で非活性にする場合

## 3. エラー定義の完全性

### 3.1 エラーで書くべき5項目

| 項目 | 内容 | 例 |
|------|------|-----|
| エラー条件 | いつエラーが発生するか | 社員コードに該当する社員がいない場合 |
| メッセージID | メッセージマスターの参照キー | MSG_00089 |
| 表示オブジェクトNo | エラーを表示する画面要素 | 2（社員コード検索） |
| 復帰動作 | エラー後に画面をどう戻すか | 検索前の状態に戻る |
| バリデーション規則 | 入力値の制約 | 半角英数字8文字以内 |

### 3.2 見落としがちなエラーケース

1. **通信エラー**: API呼び出し失敗時のUI表示
2. **タイムアウト**: 長時間応答がない場合
3. **権限エラー**: API呼び出し時の権限不足
4. **データ不整合**: 取得データに矛盾がある場合
5. **ファイルバリデーション**: アップロード時のサイズ/形式チェック
6. **同時編集**: 他ユーザーが編集中の場合

## 4. API記述の精度

### 4.1 リクエストパラメーターの書き方

最低限:
```
E列: 社員コードリスト    R列: 表示対象社員の社員コードリスト
```

理想的:
```
E列: 社員コードリスト（syainCodeList）  R列: 表示対象社員の社員コードリスト
  型: Array<String>  形式: ["ABC001", "ABC002"]  必須: はい
```

### 4.2 レスポンスマッピングの書き方

画面項目（E列）とレスポンスパラメーター（R列）の対応を明記する。
補足条件はAA列に「※」付きで記載:
```
E列: 項目名    R列: 現職エリアのタイトル    AA列: ※空白フラグが「1」の場合、表示しない
```

### 4.3 API名の二重定義に注意

外部設計（画面レイアウト）ではAPI IDで参照:
```
現職エリアの表示API(id：gensyokuArea/showBlock)
```

内部設計（処理詳細）では日本語名で定義:
```
API名: (共通)現職エリア-表示
```

両者の対応を混乱させないこと。

## 5. アンチパターン集

### 5.1 「帳票を取得する」型（動作が曖昧）
問題: 何の帳票？どのAPIで？どこに表示？
改善: 「押下すると、{帳票名}のPDFをダウンロードする。ダウンロードAPI(id: XXX/report)を呼出す」

### 5.2 「処理を行う」型（他文書に丸投げ）
問題: 読者はこの設計書だけで実装できない
改善: この設計書内で概要を書き、詳細は参照先を明記する
例: 「※登録ボタンの処理について、各機能の外部設計書を参照する」← 参照先が具体的

### 5.3 「条件なし表示制御」型（暗黙の常時表示）
問題: 表示制御を省略すると「常時表示」なのか「書き忘れ」なのか区別できない
改善: 常時表示の場合でも「常時表示とする」と明記するか、単純ラベル等は省略可と判断基準を明確にする

### 5.4 「メッセージID only」型（エラー内容不明）
問題: MSG_00089だけでは開発者がエラーメッセージの内容を知れない
改善: メッセージIDに加えて、メッセージ概要（「該当する社員が見つかりません」等）を併記する

## 6. UI状態の定義方法

多くのUI設計書で不足しがちな「状態」の定義方法:

### 6.1 定義すべき状態

| 状態 | 記述場所 | 記述内容 |
|------|---------|---------|
| ローディング中 | 初期表示の動作 | 「API呼び出し中はローディングスピナーを表示する」 |
| 空状態 | レスポンス処理 | 「データが0件の場合、{メッセージ}を表示する」 |
| エラー状態 | エラー定義 | 「通信エラーの場合、{エラーメッセージ}を表示し、リトライボタンを表示する」 |
| 無効状態 | 表示制御 | 「{条件}の場合、非活性とする」 |

### 6.2 確認ダイアログの書き方

削除等の不可逆操作には確認ダイアログを定義する:
```
D列: 動作
H列: 削除確認メッセージ「{確認メッセージ}」を表示する
  ← セル内に確認ダイアログのボタン配置を記述
  L列: {キャンセルボタンテキスト}    Q列: {実行ボタンテキスト}
```
